Verziókezelés és Git alapismeretek

Mi az a verziókezelés?

A verziókezelés egy olyan rendszer, amely nyomon követi a fájlokban történő változtatásokat az idő során. Lehetővé teszi, hogy:

Visszakövessük a változtatások történetét
Párhuzamosan dolgozzunk ugyanazon a projekten
Visszaállítsunk korábbi verziókat
Hatékonyan együttműködjünk másokkal

Mi a Git?

A Git egy elosztott verziókezelő rendszer, amelyet Linus Torvalds készített 2005-ben. Néhány kulcsfontosságú jellemzője:

Elosztott rendszer

Minden fejlesztőnél ott van a teljes projekt története, nem csak egy központi szerveren.

Központi repo
Helyi másolat
Helyi másolat
Helyi másolat

Gyors és hatékony

A Git csak a változásokat tárolja, nem a teljes fájlokat minden verzióban.

Teljes fájl
↓
+10 sor módosítás
↓
-5 sor törlés

Biztonságos

Minden változás egyedi azonosítót kap, és a tartalom integritása garantált.

Git vs. GitHub - Mi a különbség?

Git

  • Verziókezelő rendszer
  • Lokálisan fut a számítógépeden
  • Parancssorból vagy GUI-n keresztül használható
  • Offline is működik
vs

GitHub

  • Webes szolgáltatás
  • Git repositoryk tárolása a felhőben
  • Kollaborációs eszközök
  • Issue tracking, pull requestek, stb.

Git parancsok részletesen

Gyakran használt Git parancsok

git init Alapok

Új Git repository létrehozása. Ez a parancs létrehozza a szükséges Git adatbázist és alapstruktúrát a projektedhez.

$ git init
Initialized empty Git repository in .git/
$ ls -la .git
drwxr-xr-x 2 user user 4096 Jan 27 10:00 branches
-rw-r--r-- 1 user user 92 Jan 27 10:00 config
-rw-r--r-- 1 user user 23 Jan 27 10:00 HEAD
drwxr-xr-x 2 user user 4096 Jan 27 10:00 objects
drwxr-xr-x 4 user user 4096 Jan 27 10:00 refs

Mit csinál pontosan?

  • Létrehozza a .git könyvtárat, ami tartalmazza a verziókezeléshez szükséges összes információt
  • Inicializálja az alapértelmezett 'master' vagy 'main' branchet
  • Beállítja az alap Git konfigurációt a projekthez

Hasznos kapcsolók:

  • --bare - Központi repository létrehozása munkakönyvtár nélkül (szervereken használjuk)
  • --shared - Csoportmunkához megfelelő jogosultságokkal inicializál
Fontos tudnivalók:
  • Ne hozz létre Git repot egy másik Git repon belül!
  • Ellenőrizd, hogy a megfelelő könyvtárban vagy-e az inicializálás előtt
  • A .git könyvtár törlésével teljesen elvész a verziókezelés
git config Alapok

Git beállítások kezelése. A konfigurációkat három szinten kezelheti: rendszer, globális (felhasználói) és projekt szinten.

$ git config --global user.name "John Doe"
$ git config --global user.email "john@example.com"
$ git config --list
user.name=John Doe
user.email=john@example.com
core.editor=vim

Legfontosabb beállítások:

  • user.name - Neved a commitokhoz
  • user.email - Email címed a commitokhoz
  • core.editor - Alapértelmezett szövegszerkesztő
  • core.autocrlf - Sorvége karakterek kezelése

Konfigurációs szintek:

--system

Az egész rendszerre érvényes beállítások (/etc/gitconfig)

--global

A felhasználó összes projektjére érvényes (~/.gitconfig)

--local

Csak az aktuális projektre érvényes (.git/config)

Hasznos tippek:
  • Windows felhasználóknak ajánlott: git config --global core.autocrlf true
  • Linux/Mac felhasználóknak: git config --global core.autocrlf input
git add Alapok

Fájlok előkészítése commit-hoz (staging area). Ez a parancs jelzi a Git-nek, hogy mely változtatásokat szeretnénk a következő commit-ba belevenni.

$ git add index.html
$ git add styles/
$ git add .
$ git status
Changes to be committed:
new file: index.html
modified: styles/main.css

Gyakori használati módok:

  • git add fajlnev - Egy konkrét fájl hozzáadása
  • git add könyvtár/ - Egy teljes könyvtár hozzáadása
  • git add . - Minden változtatás hozzáadása az aktuális könyvtárból
  • git add -p - Interaktív hozzáadás, ahol kiválaszthatod a változtatások részeit

Hasznos kapcsolók:

  • -u - Csak már verziókezelt fájlok hozzáadása
  • -A vagy --all - Minden változtatás hozzáadása, beleértve a törölt fájlokat is
  • -p vagy --patch - Interaktív hozzáadás
Mire figyeljünk:
  • A git add . minden változtatást hozzáad, beleértve az ideiglenes fájlokat is
  • Érzékeny adatokat tartalmazó fájlokat ne verziókezeljünk
  • Nagy bináris fájlokat érdemes kihagyni a verziókezelésből
git commit Alapok

A staging area-ban lévő változtatások véglegesítése. A commit egy "pillanatfelvétel" a projekt állapotáról, amit később visszanézhetünk vagy visszaállíthatunk.

$ git commit -m "Add navigation menu"
[main 5d6a89b] Add navigation menu
2 files changed, 35 insertions(+), 5 deletions(-)
$ git commit -am "Fix typo in header"

Commit üzenetek írása:

Egy jó commit üzenet formátuma:

Rövid (50 karakter vagy kevesebb) összefoglaló

Részletesebb magyarázó szöveg, ha szükséges. Körülbelül
72 karakternél törjük a sorokat. Bizonyos kontextusokban az
első sor a commit email tárgyaként szolgál, a többi pedig
a törzsében jelenik meg.

- Pontokba is szedhetjük a változtatásokat
- További részletek felsorolása

Hasznos kapcsolók:

  • -m "üzenet" - Commit üzenet megadása parancssorból
  • -a - Minden módosított és követett fájl commitolása (staging nélkül)
  • --amend - Utolsó commit módosítása
  • -S - Commit aláírása GPG kulccsal

Legjobb gyakorlatok:

  • Egy commit egy logikai egységet tartalmazzon
  • Írj érthető, informatív commit üzeneteket
  • Használj jelen időt az üzenetekben (pl. "Add feature" nem pedig "Added feature")
  • Ne commitolj félkész vagy nem működő kódot
  • Rendszeresen készíts commit-okat, ne várj túl sokáig
Gyakori hibák:
  • Túl sok változtatás egy commit-ban
  • Nem informatív commit üzenetek ("fix", "update")
  • Ellentmondó változtatások egy commit-ban
git push Távoli

Helyi változtatások feltöltése a távoli repository-ba. Ez a parancs szinkronizálja a helyi és távoli állapotot.

$ git push origin main
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Writing objects: 100% (3/3), 282 bytes | 282.00 KiB/s, done.
$ git push -u origin feature-branch
$ git push --force origin main

Gyakori használati módok:

  • git push origin main - Változtatások feltöltése a main branch-re
  • git push -u origin feature - Új branch feltöltése és követési kapcsolat beállítása
  • git push --force - Erőltetett feltöltés (veszélyes!)
  • git push --tags - Tag-ek feltöltése

Hasznos kapcsolók:

  • -u vagy --set-upstream - Követési kapcsolat beállítása
  • --force - Erőltetett push (óvatosan használd!)
  • --force-with-lease - Biztonságosabb erőltetett push
  • --tags - Az összes tag feltöltése
Veszélyes műveletek:

A --force kapcsoló használata veszélyes lehet, mert:

  • Felülírja a távoli történetet
  • Más fejlesztők munkáját tönkreteheti
  • Elveszhetnek commit-ok

Helyette használd a --force-with-lease kapcsolót, ami csak akkor engedi a push-t, ha senki más nem módosította a távoli branch-et.

git pull Távoli

Távoli változtatások letöltése és beolvasztása a helyi branch-be. Tulajdonképpen egy git fetch és egy git merge kombinációja.

$ git pull origin main
From github.com:user/repo
* branch main -> FETCH_HEAD
Updating abc123..def456
Fast-forward
$ git pull --rebase origin main

Pull stratégiák:

Merge (alapértelmezett)
  • Létrehoz egy merge commit-ot
  • Megőrzi a teljes történetet
  • Több merge commit keletkezhet
Rebase
  • Lineáris történetet hoz létre
  • Tisztább commit história
  • Újraírja a helyi commit-okat

Hasznos kapcsolók:

  • --rebase - Rebase stratégia használata merge helyett
  • --no-commit - Nem hozza létre automatikusan a merge commit-ot
  • --squash - Összevonja a változtatásokat egy commit-ba
Pull konfigurálása:

Beállíthatod az alapértelmezett pull stratégiát:

  • git config --global pull.rebase true - Mindig rebase-t használ
  • git config --global pull.ff only - Csak fast-forward merge-t engedélyez
git merge Branch

Két branch egyesítése. A változtatások összefésülése egy új commit-ba, megőrizve mindkét branch történetét.

$ git checkout main
$ git merge feature-branch
Updating abc123..def456
Fast-forward
$ git merge --no-ff feature-branch
Merge made by the 'recursive' strategy.

Merge típusok:

Fast-forward merge
  • Ha a main branch nem változott
  • Egyszerűen előrelépteti a main-t
  • Nem hoz létre új merge commit-ot
Recursive merge
  • Ha mindkét branch változott
  • Létrehoz egy új merge commit-ot
  • Megőrzi a teljes branch történetet

Hasznos kapcsolók:

  • --no-ff - Mindig hoz létre merge commit-ot
  • --squash - Összevonja a változtatásokat egy commit-ba
  • --abort - Megszakítja a merge-t konfliktus esetén
Merge konfliktus kezelése:
  1. A Git megjelöli a konfliktusokat a fájlokban
  2. Kézzel kell kijavítani a konfliktusokat
  3. git add a javított fájlokat
  4. git commit a merge befejezéséhez
git rebase Haladó

Branch történetének átírása, áthelyezése egy másik kiindulási pontra. A rebase "újrajátssza" a változtatásokat, létrehozva egy tisztább, lineáris történetet.

$ git checkout feature-branch
$ git rebase main
First, rewinding head to replay your work on top of it...
Applying: Add new feature
$ git rebase -i HEAD~3
pick abc123 Add navigation
squash def456 Fix navigation style
pick ghi789 Add search feature

Mikor használjuk a rebase-t?

  • Feature branch naprakészen tartása a main változásaival
  • Commit történet tisztítása merge előtt
  • Commit-ok átrendezése, összevonása vagy törlése

Interaktív rebase műveletek:

  • pick - Commit megtartása
  • reword - Commit üzenet módosítása
  • edit - Commit módosítása
  • squash - Commit összevonása az előzővel, üzenetek megtartása
  • fixup - Commit összevonása az előzővel, üzenet eldobása
  • drop - Commit eldobása
Figyelmeztetés!

A rebase veszélyes művelet lehet, mert:

  • Átírja a commit történetet
  • Ne használd olyan branch-eken, amiket mások is használnak
  • Ha már push-oltál egy branch-et, csak --force push-sal tudod frissíteni

Hibaelhárítás

Gyakori Git problémák és megoldásaik

Merge konfliktus

merge conflict
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

Megoldás lépései:

  1. Ellenőrizd az érintett fájlokat
    $ git status
  2. Nyisd meg a konfliktust tartalmazó fájlt
    
<<<<<<< HEAD
Az én változtatásom
=======
A másik fejlesztő változtatása
>>>>>>> feature-branch
  3. Javítsd ki a konfliktust

    Távolítsd el a jelölőket és hagyd meg a kívánt kódot

  4. Mentsd el a változtatásokat
    $ git add index.html
    $ git commit -m "Merge conflict resolved"

Push elutasítva

push remote
! [rejected] main -> main (fetch first)
error: failed to push some refs to 'origin'

Megoldás lépései:

  1. Először húzd le a távoli változtatásokat
    $ git pull origin main
  2. Oldd meg az esetleges konfliktusokat
  3. Próbáld újra a push-t
    $ git push origin main

Ha biztosan felül szeretnéd írni a távoli változtatásokat (nem ajánlott): git push -f origin main

Autentikációs hiba

auth ssh
Permission denied (publickey)

Megoldás lépései:

  1. Ellenőrizd az SSH kulcsokat
    $ ls -al ~/.ssh
  2. Ha nincs SSH kulcs, generálj egyet
    $ ssh-keygen -t ed25519 -C "email@example.com"
  3. Add hozzá a kulcsot a GitHub-hoz

    Másold be a publikus kulcsot (~/.ssh/id_ed25519.pub) a GitHub beállításokba

  4. Teszteld a kapcsolatot
    $ ssh -T git@github.com

Commit visszavonása

commit undo

Lehetséges megoldások:

Utolsó commit módosítása
$ git commit --amend

Ha még nem push-oltad a változtatást

Commit visszavonása új committal
$ git revert HEAD

Ha már push-oltad a változtatást

Reset a korábbi állapotra
$ git reset --hard HEAD~1

VIGYÁZAT: Végleg törli a változtatásokat!
